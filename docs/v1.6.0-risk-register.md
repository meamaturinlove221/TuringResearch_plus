# v1.6.0 Risk Register

Status: final scope risk register.

Round: 361.

v1.6 prepares the project for manual public release execution. The main risks
come from crossing from dry-run/manual-pack material into public-facing action.

| Risk | Impact | Mitigation |
| --- | --- | --- |
| docs deployment publishes private material | privacy breach | final dist scan, security/privacy gate, human approval before deployment |
| fake or stale public URL appears in docs | misleading release state | URL insertion policy; no URL until maintainer verifies it |
| GitHub Pages workflow deploys unexpectedly | uncontrolled publication | workflow remains draft/approval-gated unless explicitly enabled |
| split repo is created from unreviewed pack | privacy/compliance failure | final manual execution pack and human confirmation gate |
| child repo lacks flagship backlink | fragmented identity and star dilution | require README backlink before creation or release |
| release artifact includes raw data or restricted payloads | public leak | artifact manifest, large-file scan, privacy/security gate |
| package install metadata drifts from version docs | broken public install | release contracts and version smoke before artifact build |
| optional live smoke leaks credentials | credential exposure | env-only secrets, skipped-by-default live tests, log review |
| optional live smoke runs remote commands | remote safety failure | no remote command execution in v1.6 scope |
| demo screenshots imply observed research results | credibility and claim risk | label assets as demo/local-first; no fake/demo result promotion |
| public launch checklist is treated as publication | process confusion | checklist is manual guidance only; no tag/PyPI/release automation |
| ARIS enters implementation line | false authority and scope creep | keep ARIS deferred; require separate study-only scope lock |

## Review Requirement

Every v1.6 item that crosses from readiness into public deployment, repository
creation, external push, artifact publication, live provider use, or tag
creation needs a dedicated gate and maintainer approval.
