# TuringResearch Plus v0.1.0 Release Operation Plan

Date: 2026-05-20

This plan describes the manual release operation for TuringResearch Plus `v0.1.0`. It is a checklist and procedure only. It does not publish, push tags, create a remote release, change public API, or add features.

## Release Identity

- Project: TuringResearch Plus
- Version: `0.1.0`
- Tag suggestion: `v0.1.0`
- Release title: `TuringResearch Plus v0.1.0`
- Core package: `turing_research`
- Plus package: `turing_research_plus`
- MCP server: `turingresearch-plus`
- Release description source: `docs/release-notes-v0.1.0.md`

## Pre-Release Checks

Machine-verifiable gates:

- Tests pass.
- Docs pass name integrity checks.
- Examples fake mode pass.
- Package import pass.
- MCP smoke pass.
- Name audit pass.
- Skills integrity pass.
- Source Hygiene policy present.

Manual gate:

- License decision confirmed.

Recommended commands:

```powershell
python -m pytest
python -m ruff check .
python -m mypy src
python -m pytest tests/contract/test_name_integrity.py
python -m pytest tests/contract/test_skills_integrity.py
python -m pytest tests/workflow
python -m pytest tests/contract/test_package_imports.py tests/contract/test_entry_points.py
python -m pytest tests/contract/test_mcp_server_import.py tests/contract/test_mcp_tool_registry.py tests/contract/test_mcp_stdio_safety.py
```

## Manual Human Review

Review these before tagging:

- `README.md`
- `LICENSE`
- `docs/release-notes-v0.1.0.md`
- `docs/v0.1.0-known-issues.md`
- `docs/v0.1.0-limitations.md`
- public examples under `examples/`
- `docs/source-hygiene-policy.md`
- `.env.example` and API key handling
- no private data
- no large files accidentally committed
- issue templates and pull request template

## Release Preparation Sequence

1. Confirm the workspace is in a Git repository.
2. Confirm `VERSION` is `0.1.0`.
3. Confirm `pyproject.toml` project version is `0.1.0`.
4. Confirm package `__version__` values are `0.1.0`.
5. Confirm `CHANGELOG.md` has a `[0.1.0] - 2026-05-20` section.
6. Run the pre-release commands.
7. Review docs and public repo hygiene files.
8. Confirm license decision.
9. Create a release branch if the maintainer wants one.
10. Create local tag `v0.1.0` only after human approval.
11. Push branch, tag, or remote release only by explicit maintainer action.

## Do Not

- Do not publish automatically.
- Do not push git tags automatically.
- Do not create a remote release automatically.
- Do not change public API.
- Do not add new features.
- Do not enable default live API execution.
- Do not commit secrets, private data, private papers, or restricted datasets.

## Decision Rule

Release operation can proceed only when all machine gates pass and the human review gates are confirmed.
