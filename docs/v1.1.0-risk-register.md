# v1.1.0 Risk Register

Status: planning risk register.

Round: 199.

## Major Risks

| Risk | Severity | Mitigation |
| --- | --- | --- |
| Physical split weakens flagship positioning | high | Split only demo/case spokes first; child README must point back to flagship above the fold. |
| Case repository accidentally implies VGGT research success | high | Keep public-safe case wording strict; retain claim safety report and human-review language. |
| Examples repository creates docs drift | medium | Create examples repo only after sync policy is approved and flagship quickstart remains complete. |
| Public docs site becomes stale | medium | Generate from existing docs where possible; add post-launch docs monitoring. |
| Local server dashboard expands into SaaS scope | high | Keep local-only, no login, no cloud user system, and static fallback. |
| Paper writing beta is mistaken for automatic paper writing | high | Label as beta drafting/review aid; require evidence refs and human review. |
| Plugin ecosystem invites unsafe third-party execution | high | Keep third-party plugins disabled by default; require manifests, review, sandbox policy, and compatibility checks. |
| Live web polish introduces default networking | high | Keep fake mode default; require explicit live opt-in and no real API keys in repo. |
| Compliance helpers are interpreted as legal advice | medium | Keep disclaimers visible and require human review. |
| CI/CD polish touches release automation too broadly | medium | Start with checks and docs before publishing automation. |

## Release Blockers For Any v1.1 Item

- name integrity failure;
- privacy/security gate failure;
- public demo leakage;
- secrets detected;
- raw data or restricted model payload included;
- private local paths included;
- unsupported experiment success claim;
- fake/demo output promoted as result evidence;
- live networking required by default;
- unknown plugin execution enabled;
- child repository URL added before the repository exists.

## Watch Items

- public launch feedback patterns;
- install/quickstart failures;
- docs confusion around fake/live modes;
- split repo maintenance cost;
- star/fork/watch trend after launch;
- interview usage feedback;
- live adapter provider drift;
- plugin contribution interest.
