# v0.8.0 Sprint 1 Recommendation

Status: planning recommendation.

Round: 149.

## Recommendation

Sprint 1 should focus on **Local Server Dashboard**.

This is the strongest first v0.8 move because v0.7 already has static
dashboards, public demos, workspace exports, vault UI outputs, case-study
reports, advisor exports, and quality gates. A local dashboard can connect
those artifacts into a practical daily interface without requiring cloud
accounts, remote sync, live adapters, or experiment execution.

## Why This First

- It improves public demo usability immediately.
- It gives multi-project, compliance, vault, case-study, and export outputs a
  single local browsing surface.
- It can preserve the existing static HTML/Markdown export path.
- It creates a useful host surface for later paper beta and plugin registry
  views.
- It avoids the higher risk of runtime plugin execution.

## Proposed Sprint 1 Scope

1. Local-only dashboard server design.
2. Workspace/project navigation.
3. Evidence, artifact, route, failure, advisor, export, compliance, and vault
   panels.
4. Public demo browser.
5. Static export fallback.
6. Privacy scan and fake/demo boundary checks.
7. Local server documentation and troubleshooting.

## Not In Sprint 1

- No cloud deployment.
- No user accounts.
- No default external bind.
- No private data auto-discovery.
- No live adapter execution.
- No plugin execution.
- No paper conclusion generation.
- No remote experiment execution.

## Suggested Follow-up Sprints

| Order | Focus | Reason |
| --- | --- | --- |
| 2 | Research paper writing beta | Reuse dashboard navigation and existing paper assembly gates. |
| 3 | Public plugin registry draft | Display registry metadata and compatibility reports without enabling plugins. |
| 4 | More case studies | Feed the dashboard and public demo library with review-safe examples. |
| 5 | Real OS-level plugin sandbox research | Start after dashboard and registry metadata clarify the permission surface. |

## Sprint 1 Acceptance Questions

- Can a user browse a demo workspace locally without network access?
- Are fake/demo and not-ready states visible?
- Are privacy and compliance warnings visible?
- Does the dashboard avoid reading private paths unless explicitly configured?
- Can outputs still be exported as static HTML/Markdown?
- Do default tests avoid live services and secrets?

## Decision

Recommended Sprint 1: `local_server_dashboard`.

Recommended status: planning only until a v0.8 scope-lock round creates feature
capsules, contracts, and tests.
