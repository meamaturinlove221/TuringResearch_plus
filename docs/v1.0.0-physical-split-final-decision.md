# v1.0.0 Physical Split Final Decision

Status: final decision for v1.0 public launch.

Round: 188.

This document decides whether TuringResearch Plus should physically split
repositories before the v1.0 public launch. It does not create GitHub
repositories, move files, push external repositories, or change install paths.

## Decision

Decision: `DELAY PHYSICAL SPLIT FOR v1.0 PUBLIC LAUNCH`.

v1.0 should publish the main repository first. Physical split is not a v1.0
launch prerequisite.

## Ready-to-create Candidates

The following candidates may enter `ready-to-create-after-human-approval`:

1. `turingresearch-vggt-case`
2. `turingresearch-examples`

They are safe as future spoke repositories because they are demo/case oriented,
do not own core runtime APIs, and can point back to the flagship repository.

## Not Split In v1.0

The following remain in the flagship for v1.0:

- core
- paper
- artifact
- dashboard
- plugins

Reason: these modules still carry shared product meaning, API stability risk,
plugin safety boundaries, install behavior, and release-gate complexity.

## Main Repo Comes First

The main repository remains the only flagship:

- primary star target;
- install and quickstart entry point;
- docs index;
- public demo overview;
- release notes and go/no-go history;
- privacy/security/compliance/release gates;
- package and namespace compatibility story.

Future spoke repos should amplify the flagship, not compete with it.

## Approval Required

Creating any child repository requires:

- maintainer approval;
- final export tree approval;
- privacy/compliance/claim-safety gate;
- license review;
- README review;
- confirmation that the child repo points back to the flagship above the fold.

## Final v1.0 Position

For v1.0, the correct posture is:

- launch the main repo first;
- keep split candidates documented and dry-run-ready;
- create case/examples spokes only after public launch feedback or explicit
  maintainer approval;
- keep core runtime and package installation in the flagship.
