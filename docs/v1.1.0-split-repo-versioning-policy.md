# v1.1.0 Split Repo Versioning Policy

Status: policy.

Round: 212.

Future split repositories should remain easy to trace back to the flagship
TuringResearch source. They should not create a parallel product versioning
story.

## Version Model

Use mirror versions derived from the flagship:

- `unreleased`: local `split_ready/` bundle or unpublished draft;
- `mainline-sync`: manual sync from current flagship mainline;
- `vX.Y.Z-sync.N`: reviewed mirror of flagship `vX.Y.Z`;
- `case-revision.N`: case-study text revision when no flagship release changed;
- `demo-revision.N`: examples/demo text revision when no flagship release
  changed.

## Required Metadata

Every spoke manifest should record:

- spoke repository id;
- flagship source;
- source version or source commit;
- sync status;
- sync date;
- reviewed file set;
- safety gate status;
- human reviewer or approval note.

## What Spokes Must Not Do

- Do not invent a feature version independent from the flagship.
- Do not claim core feature support not present in the flagship.
- Do not change install path or package name.
- Do not imply the spoke is the canonical implementation.

## Release Notes

If a spoke is published, release notes should say:

```text
This is a public-safe mirror of selected TuringResearch demo/case material.
The flagship TuringResearch repository remains the source of truth for install,
docs, public API, release gates, and issue triage.
```
