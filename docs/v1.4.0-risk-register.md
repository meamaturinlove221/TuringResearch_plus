# v1.4.0 Risk Register

Status: adjusted risk register.

Round: 291.

| Risk | Impact | Mitigation |
| --- | --- | --- |
| treating human scan as strict diff | Could create fake upstream change claims. | State that Round 291 uses a human scan summary, not machine added/modified/deleted evidence. |
| production parity overclaim | Users may think fake/default parity proves live provider or remote parity. | Label fake/default, optional live, deferred, and rejected surfaces separately. |
| Session production parity drifts into remote execution | Could enable unsafe command execution or data movement. | Keep remote execution disabled by default and require explicit live opt-in for transfer-only paths. |
| Scholar production parity downloads papers automatically | Could violate provider/paywall boundaries. | Keep paper content/reference/reading review-only and no automatic full paper download. |
| Web production parity scrapes private content | Could leak private data or credentials. | Keep no-login, no-cookie, public-content-only policy and fake/default tests. |
| dotenv removal regression | Example config could accidentally require or expose local secrets. | Keep no real keys in repo and require empty placeholders in public examples. |
| yogsoth production replay becomes agent runtime | Could imply autonomous tool execution. | Keep campaign/research-engine traces deterministic and review-only. |
| ARIS scope creep | Cross-model review, proof-checker, meta-optimize, or claim audit could slip into v1.4. | Maintain ARIS blocklist and reference-only docs. |
| current mainline regression | Production polish could break v1.3 runnable fake workflows. | Run name integrity, privacy/security, parity replay, and full regression before release prep. |

## Decision

The safest v1.4 posture is production-parity-first, fake-default, and
regression-first. ARIS remains deferred.
