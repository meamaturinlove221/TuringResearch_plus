# v1.0.0 Stability Policy

Status: planning policy.

Round: 173.

This policy defines what “stable” means for TuringResearch v1.0.

## Stability Levels

| Level | Meaning |
| --- | --- |
| stable | Public API is expected to remain compatible through v1.x except for documented deprecations. |
| beta | Public enough for users, but still allowed to change with migration notes. |
| experimental | Useful but not guaranteed stable; not recommended as a long-term integration surface. |
| internal | Not public API. May change without notice. |

## v1.0 Stable Candidates

Stable candidates:

- workspace registry and project summaries;
- privacy scan report model;
- quality/regression gate report;
- project template schema;
- evidence status semantics;
- route DSL and hard gate summary models;
- plugin manifest schema;
- benchmark/replay report schema.

Beta candidates:

- artifact handoff/export metadata;
- paper scaffold and deep-review DTOs;
- dashboard/export report DTOs;
- repo split dry-run manifest.

Experimental candidates:

- optional PDF/PPTX backend rendering details;
- plugin compatibility harness internals;
- future physical split workflow;
- local server dashboard concepts.

## Compatibility Rule

`turing_research_plus` remains a compatibility namespace through v1.0. New
facade namespaces may be recommended in docs, but existing import paths should
not be removed without a deprecation window.

## Deprecation Rule

Any public API deprecation must include:

- replacement path;
- migration note;
- test coverage for old and new imports;
- release note entry;
- at least one minor release of overlap when feasible.

## Stability Boundaries

Stability does not mean:

- live providers are always available;
- optional backends are installed;
- plugin execution is safe by default;
- privacy scanning is legal compliance;
- paper scaffolds are final papers;
- demos are research results.
