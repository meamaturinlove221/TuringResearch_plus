# v1.1.0 Final Scope

Status: scope locked.

Round: 205.

v1.1 is a focused post-v1 stabilization and showcase release. It should not
expand into a broad platform rewrite. The goal is to turn the v1.0 release
candidate archive into a cleaner public baseline, then advance the highest
public-value surfaces with strict safety gates.

This document locks scope only. It does not implement features, create
repositories, publish a release, create tags, or push to a remote.

## Scope Decision

Decision: `v1.1 SCOPE LOCKED`.

v1.1 default scope:

1. v1.0 baseline stabilization.
2. `turingresearch-vggt-case` split-ready finalization.
3. `turingresearch-examples` split-ready finalization.
4. Docs site.
5. Local server dashboard.
6. Paper writing beta.
7. More public demo cases.
8. Case study gallery.
9. CI/CD release hardening.
10. v1.1 release candidate.

## Scope Principles

- Keep the main repository as the flagship.
- Stabilize branch/release baseline before visible expansion.
- Prefer public-safe examples and documentation over runtime complexity.
- Keep fake/default paths working without API keys or private data.
- Keep live adapters explicit opt-in.
- Treat paper writing as reviewable beta assistance, not final paper creation.
- Treat split repositories as human-approved release operations, not automatic
  behavior.

## What Moves First

Round 204 found the v1.0 archive usable for planning but not a confirmed public
release baseline. Therefore v1.1 starts with baseline stabilization before
physical split or dashboard work.

## Locked Workstreams

### 1. v1.0 baseline stabilization

Purpose: clean up branch, release, and carryover state so v1.1 work does not
inherit ambiguous release status.

Outputs should include branch baseline docs, clean-branch checklist, and
release-state verification.

### 2. `turingresearch-vggt-case` split-ready finalization

Purpose: prepare the first spoke repository for human-approved creation after
baseline cleanup.

This remains public-safe dogfooding material only, not VGGT experiment source.

### 3. `turingresearch-examples` split-ready finalization

Purpose: prepare the examples spoke after the case repo path is clean.

This should not remove the flagship quickstart or make the main repo feel empty.

### 4. Docs site

Purpose: make public docs easier to browse without turning the project into a
hosted SaaS surface.

Static-first documentation is preferred.

### 5. Local server dashboard

Purpose: improve local exploration and daily usability.

Constraints: local-only, no login, no cloud user system, static fallback
preserved.

### 6. Paper writing beta

Purpose: support evidence-linked paper drafts, section planning, and review
workflows.

Boundary: no automatic final paper writing.

### 7. More public demo cases

Purpose: broaden public understanding with fake/demo or public-safe examples.

Every demo must pass privacy and no-unsupported-claim gates.

### 8. Case study gallery

Purpose: organize public-safe case studies and explain what each demonstrates.

The gallery must not imply research success beyond evidence.

### 9. CI/CD release hardening

Purpose: reduce dirty-branch and release-gate friction.

Start with checks, docs, and branch discipline before publishing automation.

### 10. v1.1 release candidate

Purpose: close v1.1 with regression, privacy/security, demo, docs, and release
candidate gates.
