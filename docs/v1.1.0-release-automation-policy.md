# v1.1.0 Release Automation Policy

Status: policy draft.

Round: 226.

The v1.1 release process should use automation for checks, not for publishing.

## Allowed Automation

- Run lint.
- Run unit tests.
- Run contract tests.
- Run workflow tests in fake/default mode.
- Run name integrity.
- Run privacy/security gates.
- Run docs-site build checks.
- Verify release notes, changelog, and version files.
- Generate or validate release readiness reports.

## Disallowed Automation

- No automatic PyPI publish.
- No automatic GitHub release creation.
- No automatic tag creation.
- No live tests by default.
- No upload of secrets.
- No upload of raw data or private artifacts.
- No creation of split child repositories.
- No automatic push to external child repositories.

## Release Draft Handling

Release draft content can be prepared in `docs/`, but the actual GitHub release
must require manual review and approval.

Required manual checks:

- README is honest.
- Release notes match implemented surfaces.
- Known limitations are visible.
- Public demo is fake/demo and privacy-safe.
- Case gallery is public-safe.
- Live adapters are optional.
- Split repos are not claimed as published unless they exist.

## Secret Handling

Default CI must not require secrets. If a future live workflow needs secrets,
it must be manually triggered, clearly marked live, and excluded from release
blocking default CI.

## Failure Policy

Release blocking failures:

- name integrity failure;
- privacy/security blocker;
- public demo privacy failure;
- release contract failure;
- README overclaim;
- fake/demo output marked observed;
- unsupported experiment success claim.

Non-blocking but review-required failures:

- optional live adapter unavailable;
- docs screenshot missing;
- child repository not created;
- optional PDF/PPTX backend unavailable.
