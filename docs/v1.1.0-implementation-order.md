# v1.1.0 Implementation Order

Status: locked implementation order.

Round: 205.

This order keeps v1.1 practical. It starts with baseline cleanup, then moves to
split-readiness and public-facing improvements. Work may be re-ordered only if
a release blocker or security issue requires it.

## Ordered Plan

| Order | Workstream | Output type | Gate before exit |
| --- | --- | --- | --- |
| 1 | v1.0 baseline stabilization | docs, branch-state checks, release-state checks | name integrity, privacy/security smoke, clean branch plan |
| 2 | `turingresearch-vggt-case` split-ready finalization | split bundle review, final checklist, optional repo creation plan | privacy, compliance, claim safety, no nonexistent URLs |
| 3 | `turingresearch-examples` split-ready finalization | examples bundle review, sync policy, final checklist | public demo tests, privacy gate, no huge artifacts |
| 4 | Docs site | static-first docs plan or skeleton | docs link check, no private data, README link sanity |
| 5 | Local server dashboard | local-only dashboard plan/prototype | static fallback, no login, no cloud user system |
| 6 | Paper writing beta | beta scope, contracts, fake workflow | evidence refs, human review, no final-paper claim |
| 7 | More public demo cases | demo fixtures and docs | public demo privacy gate, no unsupported claims |
| 8 | Case study gallery | gallery docs/static index | claim safety, public-safe case boundaries |
| 9 | CI/CD release hardening | checks/docs/scripts if safe | no publish automation without approval |
| 10 | v1.1 release candidate | RC report, full regression, release docs | full tests, mypy, privacy/security, public demo, name integrity |

## Dependency Notes

- Physical split decisions should not happen before branch baseline cleanup.
- Public docs site can start before physical repo creation, but must not link
  to child repositories that do not exist.
- Local server dashboard should not become a hosted dashboard.
- Paper writing beta must inherit v1.0 evidence and claim-safety gates.
- CI/CD hardening must not publish, tag, or push without approval.
