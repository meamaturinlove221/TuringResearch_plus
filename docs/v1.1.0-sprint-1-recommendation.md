# v1.1.0 Sprint 1 Recommendation

Status: recommended sprint plan.

Round: 199.

## Recommended Sprint 1

The first v1.1 sprint should prioritize:

1. `turingresearch-vggt-case` physical split, after final human approval.
2. Public documentation site.
3. Local server dashboard planning/prototype.
4. Research paper writing beta scoping.
5. More public demo planning.

## Why This Order

### 1. `turingresearch-vggt-case` physical split

The case repo is the safest first spoke because:

- it is already represented by a public-safe export bundle;
- it is a dogfooding story, not a core runtime package;
- it can point back to the flagship and protect the main star target;
- it does not change install paths or package imports.

Execution should wait for human approval, final safety review, and actual child
repository creation permission.

### 2. Public documentation site

The docs site has high public value after v1.0 because it reduces README load
and makes the Research OS surface easier to explore. It should stay static and
source-controlled first.

### 3. Local server dashboard

The dashboard is the largest usability improvement, but it must stay local-first:

- no login;
- no cloud user system;
- no default network;
- static dashboard fallback remains supported.

### 4. Research paper writing beta

This should focus on reviewable draft scaffolds, claim gates, evidence links,
and advisor review. It must not become automatic final paper writing.

### 5. More public demos

More demos should be added only if they are fully fake/demo or public-safe. They
should make capabilities easier to understand without implying research success.

## Suggested Sprint 1 Deliverables

- final approval checklist for `turingresearch-vggt-case`;
- post-create README/linking policy for the case repo;
- static public docs site plan or first static site skeleton;
- local server dashboard design note with safety constraints;
- paper writing beta scope document;
- public demo candidate list and safety criteria.

## Sprint 1 Gates

- name integrity;
- privacy/security gate;
- public demo privacy gate;
- docs link check;
- no child repo URL until the repo exists;
- no private data;
- no unsupported claims;
- no plugin execution expansion.
