# v1.5.0 Post-release Verification

Status: manual verification plan.

Round: 358.

This plan describes what a maintainer should verify after a manually approved
v1.5.0rc0 release. It does not publish anything by itself.

## Package Verification

- Confirm the release tag points to the reviewed commit.
- Confirm `VERSION` shows `1.5.0rc0`.
- Confirm package imports report `1.5.0rc0`.
- Confirm README and release notes are visible in the published source.

## Docs Verification

- If docs are published manually, confirm the live site matches the reviewed
  `docs-site/dist/` package.
- Confirm there is no unexpected analytics.
- Confirm no private path, raw data, or credential appears in the published
  pages.
- Confirm any public URL was inserted by a maintainer intentionally, not by
  an automated dry-run.

## Split Repo Verification

- If child repositories are created manually, confirm each README links back to
  the flagship main repository.
- Confirm no raw data, private paths, credentials, or unsupported claims were
  added.
- Confirm the first release draft remains manual.

## Optional Live Verification

- Confirm live tests remain skipped by default in public CI.
- Confirm live credentials are not present in repository files, logs, docs, or
  examples.
- Confirm any live provider run is private, explicit, and separately reviewed.

## ARIS Verification

- Confirm ARIS remains deferred.
- Confirm no ARIS runtime, proof-checker, meta-optimize, cross-model review, or
  paper-claim-audit path was enabled.

## Failure Handling

If any verification step fails, stop release promotion, document the issue, and
fix it in a new reviewed change. Do not silently patch published artifacts.
