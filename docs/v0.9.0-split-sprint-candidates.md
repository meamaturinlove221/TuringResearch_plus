## Candidate 1: `turingresearch-vggt-case`

Recommended sprint type: extraction rehearsal.

Why it fits:

- strongest dogfooding story;
- high interview and portfolio value;
- low runtime coupling;
- can remain public-safe with redaction and claim guards;
- reinforces the main repo as the flagship.

Required output before a physical split:

- exact file manifest;
- public-safe case-study draft;
- redaction report;
- claim safety report;
- privacy scan report;
- compliance note;
- README that links back to the flagship.

Do not include:

- private paths;
- raw data;
- model files;
- private advisor feedback;
- unsupported experiment claims;
- SparseConv3D success claim.

## Candidate 2: `turingresearch-examples`

Recommended sprint type: demo export rehearsal.

Why it fits:

- independent demo value;
- useful for community onboarding;
- lower risk than splitting code packages;
- can host fake/demo projects without moving core runtime;
- keeps star flow toward the flagship.

Required output before a physical split:

- exact examples export manifest;
- public demo smoke test;
- privacy scan;
- no-secrets scan;
- no raw-data/model-payload scan;
- README that describes fake/demo boundaries;
- sync policy back to the main repo.

Do not include:

- private VGGT files;
- raw data;
- model files;
- API keys;
- huge artifacts;
- real private logs.

## Sprint Gate

A split sprint can start only if the main repo remains complete:

- install still works from the flagship;
- public demo remains in or clearly linked from the flagship;
- docs index still explains the whole system;
- release gates still run in the flagship;
- spoke README points back to the flagship.
