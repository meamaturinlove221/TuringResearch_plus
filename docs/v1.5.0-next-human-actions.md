# v1.5.0 Next Human Actions

Status: manual action plan.

Round: 360.

This plan describes what a maintainer should do after the v1.5 final handoff.
It does not execute release, deployment, live, split, or publication actions.

## 1. Review Release Candidate Docs

- Review `docs/v1.5.0-release-notes.md`.
- Review `docs/v1.5.0-feature-list.md`.
- Review `docs/v1.5.0-known-limitations.md`.
- Review `docs/v1.5.0-test-summary.md`.
- Review `docs/v1.5.0-upgrade-guide.md`.
- Review `docs/github-release-draft-v1.5.0.md`.
- Confirm `VERSION`, `pyproject.toml`, and package versions are `1.5.0rc0`.

## 2. Review Safety Reports

- Review `docs/v1.5.0-full-replay-report.md`.
- Review `docs/v1.5.0-security-audit.md`.
- Review `docs/v1.5.0-privacy-audit.md`.
- Review `docs/v1.5.0-secret-scan-report.md`.
- Confirm no private paths, raw data, secrets, restricted model payloads, or
  fake URLs appear in v1.5 surfaces.

## 3. Inspect Docs Deployment Dry-run

- Inspect `docs-site/dist/`.
- Inspect `docs-site/dist_manifest.yaml`.
- Confirm `docs-site/deployment_dry_run_report.md` still records no deployment
  action and no public URL.
- Decide manually whether v1.6 should perform a real docs deployment.

## 4. Inspect Split Manual Packs

- Review `split_manual/turingresearch-vggt-case/`.
- Review `split_manual/turingresearch-examples/`.
- Confirm README, manifest, git init dry-run, and release checklists are
  public-safe.
- Decide manually whether v1.6 should create child repositories.

## 5. Inspect Optional Live Policy

- Review Scholar, Web / Apify, and SFTP optional live guides.
- Confirm live tests remain skipped by default.
- Confirm no credentials are stored in the repository.
- Keep any future live run private, explicit, and separately reviewed.

## 6. Inspect Dashboard Showcase

- Open the static dashboard showcase pages locally.
- Confirm the pages do not imply deployment, live execution, or verified live
  research results.
- Use the interview view only as a project architecture and engineering
  demonstration.

## 7. Prepare a Clean Release Branch

- Create or check out a clean `release/v1.5.0-rc` branch manually.
- Commit only reviewed v1.5 changes.
- Avoid carrying unrelated historical work into the release branch.
- Rerun the v1.5 full replay, security/privacy gate, release contracts, name
  integrity, public release hygiene, and mypy before publication decisions.

## 8. Decide Release Actions

Only after review, decide manually whether to:

- create a tag;
- publish a GitHub release;
- publish a PyPI package;
- deploy docs;
- create child repositories;
- begin v1.6 work.

## 9. Keep ARIS Deferred

Do not start ARIS implementation from the v1.5 handoff. If ARIS returns, it
should be a separately scoped study-only track with its own safety gate.
