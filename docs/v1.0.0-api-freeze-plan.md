# v1.0.0 API Freeze Plan

Status: planning draft.

Round: 173.

The v1.0 API freeze should make TuringResearch usable as a public, stable
local-first Research OS without freezing unstable internals too early.

## Freeze Candidates

Freeze first:

- core workspace/project DTOs;
- privacy and quality report DTOs;
- project template schema;
- evidence status labels;
- route DSL summary schema;
- plugin manifest schema;
- benchmark/replay report schema.

Freeze after review:

- artifact audit and handoff DTOs;
- paper scaffold and deep-review DTOs;
- dashboard/export quality DTOs;
- repo split dry-run manifest.

Do not freeze yet:

- optional PDF/PPTX backend implementation details;
- plugin runtime execution details;
- future OS-level sandbox details;
- local server dashboard internals;
- physical split scripts.

## API Freeze Stages

1. Inventory all public models and helper functions.
2. Compare against module public API contracts.
3. Mark each surface stable, beta, experimental, or internal.
4. Add import compatibility tests.
5. Add contract drift tests.
6. Update docs to recommend stable namespaces only.
7. Run full regression, privacy, security, and public demo gates.
8. Freeze only after maintainer review.

## Import Policy

Docs may recommend target namespaces, but `turing_research_plus` remains
supported at v1.0. Compatibility aliases should stay until a later major
version decision.

## Freeze Blockers

- unclear model ownership;
- missing docs;
- missing tests;
- private data risk;
- unstable fake/live boundary;
- plugin execution ambiguity;
- planned result marked observed;
- old naming drift.
