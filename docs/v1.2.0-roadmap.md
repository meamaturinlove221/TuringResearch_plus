# v1.2.0 Roadmap

Status: planning draft.

Round: 230.

v1.2 should move from release-candidate hardening toward more real ecosystem
capability. The emphasis is not more documentation volume; it is turning the
v1.1 review surfaces into maintainable public assets and carefully gated
runtime improvements.

This roadmap is planning only. It does not create repositories, deploy docs,
enable live networking, run remote execution, publish packages, or claim
experiment success.

## v1.2 Direction

The recommended v1.2 theme is:

> Convert the strongest v1.1 public surfaces into real ecosystem touchpoints.

That means:

- execute the safest physical split only after human approval;
- deploy docs only after the local docs builder and privacy checks remain clean;
- polish local dashboard usability without turning it into SaaS;
- prototype Pod Lifecycle Manager and Context Return Verifier as safety-first
  local tooling, not remote execution orchestration;
- refine paper writing beta around evidence and human review, not final paper
  generation.

## Recommended Priority

1. Physical split execution for `turingresearch-vggt-case` and
   `turingresearch-examples`.
2. Public docs deployment.
3. Local server dashboard polish.
4. Pod Lifecycle Manager.
5. Paper writing beta refinement.

## Sequencing

1. Human approval for child repo creation and final safety checks.
2. Create `turingresearch-vggt-case`; update flagship links only after the repo
   exists.
3. Create `turingresearch-examples` if the case repo validates the sync pattern.
4. Prepare public docs deployment with no private data and no fake links.
5. Polish local server dashboard UI and route coverage while keeping localhost
   only.
6. Scope and prototype Pod Lifecycle Manager as local validation, not remote
   execution.
7. Refine paper writing beta and evidence/claim/citation gates.
8. Consider live web/scholar polish and Apify templates only after fake/default
   regression remains stable.

## Release Gates

Every v1.2 item must preserve:

- no private data;
- no secrets;
- no raw data;
- no restricted model payloads;
- no unsupported experiment success claims;
- no fake/demo output promoted to observed evidence;
- no automatic experiment execution;
- no automatic final paper writing;
- live features explicit opt-in only;
- unknown plugin execution disabled;
- flagship repository remains the main install and star target.

## Expected v1.2 Outcome

The ideal v1.2 result is a more credible ecosystem shape:

- one or two real child repos created manually and safely;
- a public docs deployment generated from reviewed docs;
- a more useful local dashboard;
- first local-only pod lifecycle validation primitives;
- a stronger paper writing beta;
- no cloud/SaaS overreach.
