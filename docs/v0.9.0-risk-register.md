# v0.9.0 Risk Register

Status: planning risk register.

Round: 167.

## Physical Split Risks

| Risk | Severity | Mitigation |
| --- | --- | --- |
| Star attention is scattered across small repos | high | Keep main repo as flagship; every spoke links back to it. |
| Core split makes flagship feel empty | high | Do not split core in v0.9. |
| Paper/artifact APIs are unstable | high | Keep them in monorepo until beta/stable contracts exist. |
| Case study leaks private paths or sensitive details | high | Require redaction, privacy scan, compliance review, and claim guard. |
| Examples repo accidentally packages raw data | high | Use exact export manifests and public-demo privacy gate. |
| Plugin repo invites unsafe third-party expectations | high | Defer plugin split until third-party ecosystem and review harness mature. |
| Extra repos increase maintenance cost | medium | Limit v0.9 to case/examples rehearsal only. |
| Users confuse demos with research results | medium | Label all examples as fake/demo and keep observed evidence rules explicit. |
| License overclaim in case/examples | medium | Require compliance disclaimer and human review. |
| Main repo loses complete demo path | high | Keep flagship demo and docs complete before any split. |

## Release Blockers

Any of these blocks a physical split:

- secrets detected;
- private paths detected;
- raw data included;
- SMPL-X or other restricted model payload included;
- unsupported VGGT or SparseConv3D success claim;
- fake/demo result marked observed;
- missing flagship back-link;
- missing independent smoke test;
- missing maintainer approval.

## Operational Risks

- duplicated docs drift between main and spoke repos;
- versioning confusion;
- issue triage split across repositories;
- unclear ownership;
- users installing the wrong repo first.

Mitigation: treat v0.9 as rehearsal-first. Do not create physical repositories
until the exact export trees are frozen and reviewed.
