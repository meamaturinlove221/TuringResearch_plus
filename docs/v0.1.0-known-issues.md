# TuringResearch Plus v0.1.0 Known Issues

## Active Release Blockers

None recorded after Round 29 documentation validation.

## Known Issues And Watch Items

### Git Repository State

The local workspace used for release preparation is not currently initialized as a Git repository, so `git status` cannot be used until the project is placed in a repository.

Impact: release tagging requires repository initialization or sync before tagging.

### License Decision

`pyproject.toml` currently declares a proprietary license. An open source license has not been selected.

Impact: public distribution under an open source license requires explicit maintainer review and synchronized license updates.

### Contract-Only Tools

Some public MCP tools are `contract_only`.

Impact: these tools are documented as planned interfaces, not live or complete implementations.

### Optional PDF Dependency

Local PDF conversion through the PyMuPDF adapter requires the optional `pdf` extra.

Impact: `import turing_research.pdf` works without PyMuPDF, but conversion returns a clear converter-unavailable error when the dependency is missing.

### Fake / Dry-Run Scope

Many workflows use fake services or dry-run outputs.

Impact: these validate workflow shape and contracts, not live research completeness.

### Paper Draft Gate

Paper draft generation is blocked without `ExperimentReport`.

Impact: this is intentional and prevents fabricated experimental results.

## Not Known Issues

- Missing live API keys are not a blocker for default tests.
- Lack of real network access is not a blocker for `v0.1.0`.
- Heavy OCR and advanced PDF layout parsing are not expected in `v0.1.0`.
