# v1.2.0 Interview FAQ

Status: interview FAQ.

Round: 253.

## What is TuringResearch?

TuringResearch is a local-first Research OS. It organizes research work into
evidence, artifacts, routes, paper scaffolds, dashboards, advisor packs, plugin
policies, and release gates.

## What changed in v1.2?

v1.2 focused on original-reference parity. It aligned stable ideas from the
tracked reference repos into TuringResearch's own local-first architecture:
session context safety, scholar/web usability, MCP config, skill SOPs,
campaigns, vault/ontology, stress tests, experiment runbooks, and public demos.

## Why did you do original parity before ARIS?

Because parity work improves reliability and clarity first. It gives the system
a stronger base before adding speculative review loops. In a real engineering
setting, that reduces blast radius and makes later innovation easier to test.

## Why not add ARIS now?

ARIS-style cross-model review, meta-optimization, proof-checking, and paper
claim audit are valuable but risky. They can create opaque authority chains,
overstate correctness, or blur the line between review assistance and final
research output. v1.2 defers them to v1.3 study.

## How do you avoid over-engineering?

Each v1.2 item had to map to a stable reference idea, a local TuringResearch
equivalent, docs, tests, and safety boundaries. Heavy, live, or high-risk items
were deferred or rejected.

## How do you prove the project still runs?

The project uses contract tests, unit tests, workflow tests, name integrity,
privacy and release hygiene, focused mypy checks, ruff checks, and full fake
workflow replay. v1.2 also added a public demo refresh and Reference Parity
Dashboard.

## Does it run real experiments?

No. v1.2 creates safe runbooks, artifact requirements, and ingest contracts. It
does not automatically execute experiments, call Modal, use GPUs, or turn demo
material into verified live claims.

## Is this just prompt engineering?

No. The main work is system design: data models, contracts, workflow gates,
public demo safety, docs, MCP configuration, plugin boundaries, and regression
discipline.

## What should I show an interviewer first?

Start with the architecture story, then show the Reference Parity Dashboard and
v1.2 public demo. Use the ARIS deferral decision to demonstrate engineering
judgment: the project is ambitious, but not reckless.

## What is the strongest engineering point?

The strongest point is scope control. v1.2 deliberately chose stable,
testable parity over flashy autonomy, and preserved local-first, fake/default,
privacy-safe workflows.
