# v1.0.0 Implementation Freeze Policy

Status: freeze policy locked after upstream refresh.

Round: 178 upstream-adjusted scope lock.

This policy defines how work should proceed after v1.0 scope lock.

## Freeze Decision

After Round 178, v1.0 should enter implementation freeze for feature work.

Allowed changes:

- release blocker fixes;
- test fixes;
- documentation corrections;
- API compatibility repairs;
- privacy/security/compliance fixes;
- packaging/import fixes;
- README and quickstart clarity;
- release notes, changelog, and go/no-go updates;
- safety wording corrections for Campaign Catalog, MCP config, and Pod Context
  Lifecycle.

Not allowed without explicit scope reopen:

- new product features;
- new live adapters;
- new plugin execution paths;
- new physical split automation;
- new default network behavior;
- new final-paper writing behavior;
- new experiment execution behavior;
- remote execution orchestration;
- MinerU / heavy PDF fallback runtime;
- complex autonomous campaign runtime.

## API Freeze Rule

Stable or beta API contracts may be refined only to:

- fix inconsistency with implementation;
- document missing fields;
- reduce ambiguity;
- add compatibility aliases;
- clarify stability level.

They should not be expanded with new v1.1 or v1.2 features.

## Upstream Adjustment Rule

The Round 175-177 upstream absorption line is closed for v1.0. Further upstream
signals should be classified into v1.1/v1.2 unless they reveal a release
blocker or safety issue.

## Documentation Freeze Rule

Docs may be edited to improve accuracy, safety, installability, and release
readiness. Docs must not make roadmap items sound shipped.

## Test Freeze Rule

Tests may be added or updated for:

- release criteria;
- privacy/security gates;
- name integrity;
- contract drift;
- import compatibility;
- public demo safety;
- quality/regression gate;
- split dry-run safety;
- Campaign Catalog completeness;
- MCP fake/live defaults;
- Pod Context Lifecycle safety.

Tests should not require live networking by default.

## Release Branch Rule

Before release:

- create or switch to a clean release branch;
- stage only reviewed v1.0 files;
- avoid carrying unrelated historical worktree changes;
- run full checks from the clean branch;
- require maintainer approval before push/tag/release.

## Scope Reopen Rule

Scope can be reopened only if:

- a release blocker cannot be fixed inside current scope;
- a safety issue requires structural changes;
- a public API contract is found to be misleading;
- maintainer explicitly approves the reopen.

All reopened work must update final scope, risk register, release criteria, and
ledger entries.
