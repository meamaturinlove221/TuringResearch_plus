# v0.8.0 Roadmap

Status: planning draft.

Round: 149.

v0.8 should build on the v0.7 release-candidate posture by making the local
research OS easier to use in everyday work. The focus shifts from public
release readiness toward a more usable local dashboard, stronger paper assembly
support, safer public plugin discovery, broader demo/case-study coverage, and
research toward a real operating-system-level plugin sandbox.

This is a roadmap, not an implementation commitment.

## Roadmap Theme

v0.7 prepared controlled public-readiness surfaces: trusted plugin manifests,
policy-level sandboxing, optional advisor exports, refined static dashboards,
compliance review aids, vault browsing, case-study drafting, and expanded
public demos. v0.8 should keep those safety defaults while improving the main
daily workflow:

- local dashboards become easier to browse without turning into SaaS;
- paper-writing support moves from scaffold/checklist mode toward a beta that
  still blocks unsupported conclusions;
- public plugin registry work starts as metadata, compatibility, and safety
  review, not automatic installation;
- more case studies demonstrate reusable workflows with demo-safe or reviewed
  evidence;
- operating-system sandbox work remains research-gated until it can be tested
  without executing unknown code.

## Recommended v0.8 Priority Order

1. Local server dashboard.
2. Research paper writing beta.
3. Public plugin registry draft.
4. More case studies.
5. Real OS-level plugin sandbox research.

## Suggested Sprint Shape

### Sprint 1: Local Dashboard Foundation

- Local-only dashboard server design and threat model.
- Project/workspace navigation, search, and filter refinement.
- Public demo dashboard path that works without live services.
- Dashboard privacy gate and static export fallback.

### Sprint 2: Paper Writing Beta

- Section assembly from existing paper scaffold components.
- Claim safety, citation safety, and evidence gap gates.
- Advisor review outputs for paper readiness.
- Explicit blocking for final abstract, final results, and final conclusions
  until human review and real evidence are present.

### Sprint 3: Public Plugin Registry Draft

- Registry metadata format.
- Compatibility and sandbox-policy labels.
- Disabled-by-default plugin discovery examples.
- Documentation for what the registry does and does not guarantee.

### Sprint 4: Case Study Expansion

- Additional public demo case studies.
- Redaction and compliance review reports.
- Case-study quality gates that block unsupported results.
- Clear distinction between fake/demo case studies and evidence-backed public
  case studies.

### Sprint 5: OS-level Sandbox Research

- Threat model for plugin execution.
- Candidate local isolation strategies.
- Test harness design for sandbox behavior.
- Decision gate before runtime loading of unknown third-party code.

## Human Review Required

- Any paper conclusion or contribution claim.
- Any promotion of planned or demo evidence to observed evidence.
- Any plugin registry inclusion.
- Any sandbox policy exception.
- Any case-study claim involving real project outcomes.
- Any dataset, model, paper figure, or code license interpretation.
- Any live adapter, remote execution, or network-enabled workflow.

## Privacy / Data / License Risk

- Local dashboards can expose private paths if they index too broadly.
- Paper-writing beta can overstate incomplete evidence if gates are weak.
- Public plugin registries can create supply-chain and trust confusion.
- Case studies can leak private artifacts or advisor feedback if redaction is
  incomplete.
- Compliance expansion can be misread as legal advice.
- OS-level sandbox research can create false confidence if policy and runtime
  enforcement are not clearly separated.

## Cannot Be Fully Automated

- Legal/license approval.
- Human paper review.
- Final paper conclusions.
- Evidence promotion to observed.
- Advisor approval.
- Plugin trust decisions.
- Case-study publication approval.
- Experiment success claims.

## Release Posture

v0.8 should remain local-first, fake/demo-first, optional-live, and
review-first. Any local server should bind to local use only by default. Any
plugin registry should remain discovery/review metadata unless a later scope
lock explicitly approves safe runtime behavior.
