# v1.0.0 Split Next Actions

Status: post-v1.0 action plan.

Round: 193.

This plan describes what to do after the flagship repository is publicly
launched. It is not an instruction to create repositories during v1.0
prelaunch.

## After Public Launch

1. Review public feedback on the flagship README, quickstart, and demos.
2. Confirm whether a separate dogfooding case repository would improve
   discovery without weakening the flagship.
3. If yes, create `turingresearch-vggt-case` from the reviewed local bundle.
4. Update the flagship README only after the repository exists.
5. Re-run privacy, compliance, claim-safety, and release-link checks.

## Second Candidate

Create `turingresearch-examples` only after:

- the case repo proves useful;
- the flagship quickstart remains clear;
- demo maintenance cost is understood;
- sync policy between flagship and examples repo is approved.

## Plugin Candidate

Delay `turingresearch-plugins` until:

- real third-party plugin interest exists;
- plugin review workflow has maintainers assigned;
- no-execute-code and no-secrets defaults are still enforced;
- contribution templates and compatibility reports are ready.

## Modules To Keep In Flagship

Do not split these during v1.0:

- core;
- paper;
- artifact;
- dashboard;
- plugin framework implementation.

## Documentation Updates When A Repo Exists

Only after a real repository exists:

- add its URL to README;
- add it to `docs/future-split-repos.md`;
- update `docs/split-ready-bundles.md` to mark the bundle as published;
- add post-create validation results;
- confirm the child README points back to the flagship.
