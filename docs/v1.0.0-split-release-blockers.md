# v1.0.0 Split Release Blockers

Status: blocker list for physical split only.

Round: 193.

These blockers apply to real split repository creation. They do not block the
main repository v1.0 public launch.

## Physical Split Blockers

| Blocker | Applies to | Status |
| --- | --- | --- |
| Maintainer approval missing | all split candidates | blocking |
| External repository creation approval missing | all split candidates | blocking |
| Final license review not recorded for real repo | all split candidates | blocking |
| Real issue/PR template plan not reviewed | all split candidates | blocking |
| Post-create link policy not executed | all split candidates | blocking |
| Main repo launch feedback not collected | case/examples | blocking before v1.0 launch |
| Real third-party plugin demand not proven | plugins | blocking |

## Not Blockers For Main Repo Launch

- Local `split_ready/` bundles existing but unpublished.
- No external child repository URL.
- No physical split before v1.0.
- Keeping core, paper, artifact, dashboard, and plugin framework in the
  flagship.

## Safety Blockers

Any future split is blocked if a bundle contains:

- private data;
- raw data;
- restricted model payloads;
- private local paths;
- real credentials;
- unsupported observed-result claims;
- fake/demo output presented as observed evidence;
- plugin examples that enable unknown code execution or secrets access.

## Release Position

The correct v1.0 release posture is:

- launch the flagship repository;
- keep split bundles local and documented;
- require human approval before any child repo creation;
- avoid splitting modules that still define the product core.
