# v0.6 Developer Guide

Status: developer consolidation guide.

Round 123 documents how to work with the v0.6 surface without adding new
runtime behavior.

## Repository Layout

- `src/turing_research/`: core package and MCP smoke server.
- `src/turing_research_plus/`: research workflow modules.
- `contracts/`: YAML contracts and safety boundaries.
- `docs/`: user, developer, design, and release-gate docs.
- `examples/`: fake/demo fixtures and dogfooding outputs.
- `lanes/`: implementation ledgers.
- `.agents/skills/`: local `turingresearch-*` skill entries.

## Development Loop

Use focused tests while editing, then run broader gates:

```powershell
python -m pytest tests/workflow/test_v0_6_full_fake_replay.py tests/contract/test_v0_6_contracts.py -q
python -m pytest tests/contract/test_name_integrity.py -q
python -m pytest tests/unit/test_quality_metrics.py tests/unit/test_regression_gate.py tests/unit/test_quality_report.py tests/workflow/test_quality_gate_public_demo.py -q
python -m mypy src
```

## Contract Rules

New v0.6 contracts should state:

- `project: TuringResearch Plus`
- `mcp_server: turingresearch-plus`
- `network_behavior: no_network`
- `requires_human_review: true`
- fake/default or review-only behavior when applicable

Do not expose new public MCP APIs without contract and namespace tests.

## Docs Rules

- Prefer linking from `docs/docs-index.md` instead of duplicating full content.
- Keep live features marked optional.
- Mark demo/fake examples explicitly.
- Keep VGGT as a dogfooding case.
- Do not write planned work as observed evidence.
- Do not add private paths, secrets, raw data, or model payloads.

## Testing Rules

- Default tests must not require network access.
- Live tests must be explicitly marked and skipped by default.
- Fake adapters must remain available for live-capable surfaces.
- Regression gates should fail on old naming, secrets, missing demos, contract
  drift, fake observed results, and live tests required by default.

## Release Rules

v0.6 docs consolidation is not a release. A future release candidate should
still require:

- clean branch review;
- full test suite;
- name integrity;
- privacy gate;
- quality regression gate;
- maintainer approval.
