# v1.1.0 Split Repo Sync Policy

Status: policy.

Round: 212.

This document defines how the flagship TuringResearch repository and future
split repositories should stay aligned after physical split. It does not
implement automatic sync, create repositories, or push remotes.

## Source Of Truth

The flagship TuringResearch repository is the source of truth for:

- install path;
- package metadata;
- public API;
- release notes;
- quickstart;
- safety policies;
- privacy and compliance gates;
- public demo baseline;
- split-ready bundle manifests.

Future split repositories are mirrors for public demo or case material only.
They must not introduce core framework behavior independently.

## Repository Roles

| Repository | Role | Sync Status |
| --- | --- | --- |
| `turingresearch` | flagship source of truth | canonical |
| `turingresearch-vggt-case` | public-safe dogfooding case mirror | ready after human approval |
| `turingresearch-examples` | public demo and template mirror | ready after human approval |
| `turingresearch-plugins` | plugin policy and registry draft mirror | deferred until ecosystem demand |

## Sync Frequency

- Before every public release candidate: review all split-ready manifests.
- After a public README or quickstart change: review spoke README crosslinks.
- After public demo changes: refresh examples mirror candidates.
- After case-study or claim-safety changes: refresh case mirror candidates.
- After security/privacy policy changes: re-run safety review before copying.
- Monthly during active v1.1 development: check drift manually.

## Versioning

Future spoke repositories should use source-derived versions:

- `mainline-sync`: current manual sync from flagship, no release guarantee;
- `vX.Y.Z-sync.N`: reviewed mirror of a flagship release;
- `unreleased`: local bundle or draft material.

The spoke version must point to the flagship version or commit used as source.
Spokes should not independently advance feature versions.

## README Backlinks

Every spoke README must:

- link or placeholder-link to the flagship above the fold;
- ask readers to use and star the flagship repository;
- state that the spoke is demo/case material only;
- state that install, docs, release, quickstart, and public API remain in the
  flagship;
- avoid nonexistent GitHub URLs before physical creation.

## Issue Backflow

Issues opened in future spoke repositories should be triaged as:

- content issue in spoke: fix in spoke and record source update if needed;
- core framework issue: redirect or mirror to flagship issue tracker;
- privacy/security issue: handle through flagship security/privacy policy;
- plugin safety issue: route to flagship plugin review policy;
- unsupported claim issue: fix both spoke text and flagship source docs.

Spoke repositories should not create independent support processes before
maintainer capacity exists.

## Safety Check Flow

Before every manual sync:

1. Confirm the source files come from the flagship repository.
2. Confirm no raw data, private paths, secrets, restricted model files, huge
   artifacts, or private logs are present.
3. Confirm fake/demo output is not described as observed research evidence.
4. Confirm README still routes readers back to the flagship.
5. Confirm manifest and safety report are updated.
6. Run split safety tests and privacy gate.
7. Require human approval before any external repository update.

## Non-goals

- No automatic sync.
- No external push.
- No repository creation.
- No independent core feature development in spokes.
- No default networking.
- No private data transfer.
