# v1.0.0 Release Criteria

Status: criteria locked after upstream refresh.

Round: 178 upstream-adjusted scope lock.

This document defines what must be true before TuringResearch Plus can be
treated as a v1.0 release candidate.

## Go Criteria

v1.0 can move to release-candidate review only if all of the following pass:

- full test suite passes;
- `python -m mypy src` passes or documented exceptions are approved;
- package import and public import tests pass;
- module public API contract tests pass;
- Campaign Catalog tests pass;
- MCP fake/live config tests pass;
- Pod Context Lifecycle safety tests pass;
- name integrity passes;
- privacy/security gate passes;
- compliance gate passes;
- quality/regression gate passes;
- public demo tests pass;
- plugin safety and compatibility tests pass;
- split dry-run safety checks pass if split artifacts are referenced;
- docs index and README links are reviewed;
- release notes, changelog, and upgrade notes are current;
- license posture is reviewed;
- maintainer approves the release.

## Content Criteria

The public package must include:

- honest README first screen;
- local install and quickstart;
- public demo suite;
- VGGT public-safe dogfooding case;
- capability overview;
- Campaign Catalog documentation;
- MCP fake/live config guide;
- Pod Context Lifecycle Safety Plan;
- plugin safety explanation;
- privacy and compliance disclaimers;
- known limitations;
- v1.0 stability policy;
- release blockers and go/no-go report.

## Safety Criteria

The release must not include:

- secrets;
- API keys;
- private local paths;
- raw data;
- SMPL-X or other restricted model payloads;
- huge private artifacts;
- fake results marked observed;
- unsupported VGGT or SparseConv3D success claims;
- unknown plugin execution enabled by default;
- live networking required by default;
- remote execution orchestration enabled by default.

## API Criteria

Before v1.0:

- stable surfaces must have contracts and tests;
- beta surfaces must be clearly labeled;
- experimental/internal surfaces must not be presented as stable;
- `turing_research_plus` compatibility namespace must remain available;
- deprecations must include replacement and migration notes.

## Upstream Adjustment Criteria

The Round 175-177 upstream absorption line is accepted only as:

- documentation and safety-plan improvements;
- fake/live MCP config clarity;
- local deterministic campaign routing metadata;
- pod context lifecycle preflight and return verification reports.

It does not authorize copying upstream code, enabling default live networking,
adding remote execution, or expanding v1.0 into heavy paper ingestion.

## Documentation Criteria

Docs must explain:

- what v1.0 is;
- what v1.0 is not;
- fake/live boundary;
- planned vs observed boundary;
- human review requirement;
- plugin safety boundary;
- optional split repo status;
- privacy/compliance limitations.

## No-go Criteria

Any of the following blocks v1.0:

- full tests fail;
- mypy fails without approved exception;
- privacy/security gate fails;
- compliance gate fails;
- public demo contains private data;
- old project name appears in release surfaces;
- fake/demo result is marked observed;
- live services become required by default;
- unknown plugin execution is enabled;
- README overclaims automatic science, final paper writing, or VGGT success;
- maintainer approval is missing.

## Release Decision

v1.0 is not automatically released by meeting technical criteria. The final
decision still requires human release review.
