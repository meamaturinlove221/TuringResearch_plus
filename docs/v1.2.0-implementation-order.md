# v1.2.0 Implementation Order

Status: scope locked.

Round: 231.

This order favors stable reference parity first. Physical split, public docs
deployment, dashboard polish, and paper beta refinement remain useful, but they
should follow or run alongside parity only when they do not disrupt the main
reference-parity path.

## Recommended Order

1. Upstream strict diff baseline.
2. MCP config parity.
3. Neocortica-Session parity.
4. Neocortica-Scholar parity.
5. Neocortica-Web parity.
6. yogsoth-ai campaign / vault / ontology parity.
7. Skill SOP parity.
8. Public demo refresh.
9. v1.2 full regression.
10. v1.3 ARIS study roadmap.

## Detailed Order

### 1. Upstream strict diff baseline

Create a strict snapshot/reporting gate that distinguishes:

- current local baseline;
- operator-supplied manual snapshot;
- live upstream scan when explicitly allowed;
- unknown or unverified changes.

Do not claim a live diff when no live scan occurred.

### 2. MCP config parity

Normalize `.mcp.example.json`, MCP docs, fake/live mode docs, and env block
policy against the stable reference patterns. Keep real keys out and live tests
disabled by default.

### 3. Neocortica-Session parity

Implement only local safety parity first:

- context lifecycle state;
- preflight checks;
- transfer policy;
- return manifest verification;
- path traversal / dotfile / shell metacharacter awareness.

No SSH, tmux, Modal, or remote command execution.

### 4. Neocortica-Scholar parity

Refine scholar pipeline docs and tests:

- MCP config examples;
- paper source fallback policy;
- citation status guard;
- heavy PDF fallback as research-only.

### 5. Neocortica-Web parity

Refine web adapter parity:

- optional live mode docs;
- Apify template planning;
- fake/default tests;
- no default live networking.

### 6. yogsoth campaign / vault / ontology parity

Gate campaign catalog, vault graph, ontology SOPs, stress-test, and experiment
planning against stable yogsoth-style concepts without implementing a new agent
runtime.

### 7. Skill SOP parity

Normalize skill entry, routing, SOP docs, and tests so public workflows have
clear skill handoffs.

### 8. Public demo refresh

Refresh demos only after parity boundaries are clear. All demos remain fake or
public-safe and must not mark fake output as observed evidence.

### 9. Full regression

Run full pytest, mypy, name integrity, privacy/security, plugin safety, docs,
dashboard, paper beta, and parity gates.

### 10. v1.3 ARIS study roadmap

Write the ARIS roadmap after parity is complete. Keep it study-first and do not
backport ARIS features into v1.2.
