# v1.0.0 Risk Register

Status: planning risk register.

Round: 173.

## Major Risks

| Risk | Severity | Mitigation |
| --- | --- | --- |
| Stable API freezes weak abstractions | high | Freeze only reviewed core surfaces; keep experimental modules clearly labeled. |
| README overclaims research automation | high | Keep human-review, fake/demo, and no-final-paper boundaries visible. |
| Public demo leaks sensitive data | high | Run privacy/security/compliance gates before release. |
| Fake result becomes observed evidence | high | Enforce evidence status tests and wording checks. |
| Plugin manifest stability implies runtime safety | high | Keep plugin execution disabled/review-gated unless sandbox is approved. |
| Split repos scatter stars | medium | Keep main repo flagship; spokes link back above the fold. |
| Optional backends become perceived requirements | medium | Keep PDF/PPTX/live dependencies optional and gracefully skipped. |
| License posture remains unclear | high | Require maintainer license review before public release. |
| v1.0 release branch includes historical dirty worktree | high | Require clean branch review and selective commit discipline. |

## Release Blockers

- full tests fail;
- mypy fails without documented exception;
- privacy/security gate fails;
- compliance gate fails;
- public demo fails;
- old name appears;
- secrets detected;
- raw data included;
- SMPL-X/model payloads included;
- private paths included;
- unsupported VGGT or SparseConv3D success claim;
- plugin execution enabled by default;
- live networking required by default;
- maintainer approval missing.

## Residual Risks After Release

- external users may misunderstand fake/demo boundaries;
- optional live providers may drift;
- future split repos may create docs drift;
- stable APIs may need deprecation paths sooner than expected.
