# TuringResearch Plus v0.2.0 Sprint 2 Integration Report

Round 50 checks the Sprint 2 data flow across route planning, hard gate
validation, failure attribution, paper method extraction, and architecture
diagram export.

This is an integration gate. It does not run VGGT, does not read `D:/vggt`,
does not call image models, does not generate real paper conclusions, and does
not implement Future Sync Adapters.

## Scope Checked

| Item | Integration status | Notes |
| --- | --- | --- |
| Experiment Route DSL | pass | Loads the Modal SparseConv3D route fixture and compiles it into a structured route plus controller prompt draft. |
| Hard Gate Library | pass | Validates route inputs with conservative `not-enough-evidence` outcomes when real backend and visual proof are absent. |
| Failure Taxonomy Engine | pass | Classifies fixture-backed missing visual proof and missing asset failures into normalized attribution reports. |
| Paper-to-Method Card | pass | Converts fake/manual NeuralBody and HumanRAM notes into method cards with VGGT mapping and human-review limitations. |
| Figure-to-Architecture | pass | Converts method cards and route specs into Mermaid, DOT, and Markdown architecture drafts. |

## Route Flow

`ExperimentRouteSpec`
-> `HardGateValidationReport`
-> `ControllerPromptDraft`

The Modal SparseConv3D route fixture is explicitly marked `planned`,
`requires-real-experiment`, and `not executed by TuringResearch`. Route compile
is a planning operation only. It is not VGGT execution, Modal execution, or proof
that SparseConv3D has succeeded.

## Failure Flow

`EvidenceLedger / ArtifactAuditReport`
-> `FailureAttributionReport`
-> `NextAction`

The integration test exercises missing visual board proof and missing asset
language. Failure attribution reports include EvidenceRef-compatible context or
set `requires_human_review`. Hard gate failures are plan validation failures;
they are not automatically experiment run failures.

## Paper Method Flow

`Paper notes / PDF markdown fixture`
-> `PaperMethodCard`
-> VGGT mapping

The NeuralBody and HumanRAM fixtures are `fake-or-manual-note` inputs. They do
not claim complete paper reading and do not fabricate citations. Method cards
preserve `requires_human_review` until real paper review evidence exists.

## Architecture Flow

`PaperMethodCard / ExperimentRouteSpec`
-> `ArchitectureDiagramSpec`
-> Mermaid / DOT / Markdown

Architecture diagrams are text-only drafts. Fixture-derived diagrams are marked
`derived-from-fixture` and `requires-human-review`. Mermaid output uses
`flowchart TB` with subgraphs; DOT export is a minimal Graphviz representation.

## Advisor Linkage

`Route + Failure + Method + Architecture`
-> can be referenced by Advisor Pack

Round 50 verifies that route IDs, failure IDs, method-card IDs, and architecture
diagram IDs can be referenced together. It does not regenerate a complete
Advisor Pack, because Sprint 2 only needs to prove referenceability from the new
modules back into the Sprint 1 advisor surface.

## Cross-Cutting Checks

- Outputs serialize to JSON through Pydantic `model_dump(mode="json")`.
- Outputs render Markdown where applicable.
- Mermaid output starts with `flowchart TB` and includes subgraphs.
- Route compile is not experiment execution.
- Hard gate failure is plan validation, not a real experiment failure.
- Failure attribution is evidence-backed or human-review gated.
- Paper method fixtures do not claim full paper review.
- Architecture diagram fixtures are derived and human-review gated.
- Proposed Sprint 2 tools remain capsule-local and are not frozen public MCP API.
- Future Sync Adapters remain non-goal.
- Historical pre-rename project naming is allowed only in migration checkpoint docs.
