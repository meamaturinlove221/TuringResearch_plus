# v1.0.0 Interview FAQ Final

Status: final interview FAQ.

Round: 185.

## 1. 这个项目解决什么问题？

它解决研究项目状态分散的问题。很多研究不是输在模型本身，而是 evidence、artifact、route、
failure、paper notes、advisor communication 和 dashboard 各自漂移。TuringResearch Plus 把这些状态
组织成本地、可审查、可回放的 workflow。

## 2. 和普通文献总结器有什么区别？

普通文献总结器主要围绕 paper summary。TuringResearch Plus 更像 Research OS：它覆盖 evidence
ledger、artifact audit、route DSL、failure taxonomy、paper scaffold、advisor pack、dashboard、
plugin safety、privacy gate 和 benchmark replay。它处理的是研究操作系统层面的状态管理。

## 3. 为什么要 Evidence Ledger？

因为研究里最危险的问题之一是 planned 变成 observed。Evidence Ledger 强制记录 claim、status、
source reference、limitations 和 human review boundary，让缺失证据不会被漂亮语言盖住。

## 4. 为什么要 Artifact Auditor？

论文和汇报不能只看文字，还要看支撑它的文件是否存在、是否完整、是否可公开、是否只是
placeholder。Artifact Auditor 用来暴露这些缺口，避免 artifact 不完整却支撑过强 claim。

## 5. 为什么要 Route DSL？

实验路线如果只在聊天记录或脑子里，就很难复盘。Route DSL 把 planned route、hard gates、failure
mode 和 next action 变成结构化对象。这样 blocked route 也能成为有价值的工程状态。

## 6. 怎么保证不伪造结果？

靠多层边界，而不是一句提示词：

- evidence status 区分 planned、fake-data、not-enough-evidence、observed；
- result table guard 阻止缺真实结果时生成数值表；
- paper assembly gate 阻止 final abstract / conclusion；
- claim guard 阻止 unsupported claims；
- privacy/compliance gate 阻止公开材料泄漏；
- fake/live boundary 保证默认测试不依赖 live adapters。

## 7. 怎么做测试？

测试分层包括：

- unit tests；
- workflow tests；
- contract tests；
- namespace import compatibility tests；
- public demo tests；
- privacy / hygiene gates；
- benchmark replay；
- mypy；
- ruff。

默认路径不需要 API key、live network、Modal、远程机器或私有项目目录。

## 8. 为什么先 monorepo？

因为主仓是旗舰入口。README、quickstart、public demo、tests、release gate 和 star 获取需要集中在一个
地方。过早拆仓会让用户不知道从哪里开始，也会削弱主仓价值。

## 9. 什么时候拆仓？

只有模块具备稳定 API、完整 docs、独立 demo、独立 tests、无私有数据、无 license risk、主仓仍然能
清晰运行时，才考虑拆。优先候选是 case study 和 examples，不是 core。

## 10. 这个项目体现了什么工程能力？

它体现的是系统设计、模块边界、测试策略、隐私安全、插件架构、文档工程、release engineering、
产品定位，以及把复杂研究流程变成可审查系统的能力。

## 11. 为什么说不是简单 prompt 工程？

因为核心工作不是写一个提示词生成文本，而是建立一套有 contract、schema、tests、privacy gate、
plugin safety、namespace compatibility、fake/live boundary 和 replay gate 的系统。prompt 只是用户
交互的一层，真正的价值在状态模型和验证链路。

## 12. VGGT dogfooding 说明了什么？

VGGT dogfooding 说明这个系统能帮助组织真实风格的研究 workflow：north star、route changes、
failures、evidence management、advisor pack、paper scaffold 和 public-safe case study。它不说明实验
已经成功，也不替代人工研究判断。
