# Project Long Pitch

Status: final portfolio draft.

Round: 174.

TuringResearch Plus is a local-first Research OS for evidence-led research
workflows.

The project starts from a practical problem: research state is usually
scattered. Papers live in one place, experiments in another, artifacts in a
folder, advisor feedback in messages, and claims in a half-written draft. That
makes it easy to lose track of what is planned, what is observed, what is
blocked, and what still needs human review.

TuringResearch Plus organizes that state into explicit local artifacts:
evidence ledgers, artifact audits, route DSL specs, failure taxonomies, paper
scaffolds, advisor packs, static dashboards, plugin manifests, privacy scans,
compliance notes, and replay gates.

It is not designed to replace scientific judgment. The point is the opposite:
make uncertainty visible. Planned work stays planned. Missing evidence stays
missing. Fake/demo examples stay fake/demo. Live adapters are optional. Unknown
plugins are not executed. Paper scaffolds do not become final paper claims.

The project has been dogfooded through a VGGT human-prior case study, but that
case is presented carefully: it demonstrates workflow discipline, not final
research success.

For portfolio review, the project shows system design, testing discipline,
privacy and safety thinking, modular architecture, plugin policy, and practical
public positioning.

## 3-minute Version

The project starts from a practical research problem: important state is
usually scattered across papers, notes, run logs, artifact folders, dashboards,
and advisor messages. Once that happens, it becomes too easy to confuse a plan
with a result, a demo with evidence, or a missing experiment with a claim.

TuringResearch Plus turns that state into local, reviewable artifacts. The core
surfaces are evidence ledgers, artifact audits, route DSL specs, failure
taxonomy, paper scaffolds, advisor packs, static dashboards, plugin manifests,
privacy scans, compliance reports, and replay gates.

The engineering focus is boundary discipline. The system does not execute
unknown plugins by default, does not require live network access for demos,
does not write final paper conclusions, and does not treat VGGT dogfooding as
experiment success. That refusal to overclaim is part of the product.

For an internship interview, the project demonstrates architecture design,
contract testing, modular monorepo evolution, privacy and safety review,
public demo packaging, and the judgment to keep ambitious tooling honest.
