# v1.0.0 Namespace Compatibility Report

Status: compatibility gate draft.

Round: 177.

This report verifies the v1.0 namespace compatibility posture. It does not move
implementation code, remove old imports, split repositories, or change package
entrypoints.

## Decision

Namespace compatibility status: `pass-with-review`.

The legacy compatibility namespace remains available:

- `turing_research_plus`

The new facade namespaces remain available:

- `turing_research_core`
- `turing_research_paper`
- `turing_research_artifact`
- `turing_research_experiment`
- `turing_research_dashboard`
- `turing_research_plugins`
- `turing_research_cases`

## Compatibility Model

The v1.0 namespace model is facade-first:

1. `turing_research_plus` remains the compatibility layer.
2. New domain namespaces provide public facade/re-export surfaces.
3. Implementations still largely live under `turing_research_plus`.
4. Docs recommend new domain namespaces for public API examples.
5. Old imports must not be removed before v1.0 release hardening completes.

## Required Checks

- `import turing_research_plus`
- `import turing_research_core`
- `import turing_research_paper`
- `import turing_research_artifact`
- `import turing_research_experiment`
- `import turing_research_dashboard`
- `import turing_research_plugins`
- `import turing_research_cases`
- public API modules expose `NAMESPACE`
- public API modules expose `COMPATIBILITY_NAMESPACE`
- compatibility aliases resolve back to `turing_research_plus` where relevant
- package discovery includes new namespace packages
- old package import tests still pass

## Boundary

Passing this gate does not mean:

- implementation has moved out of `turing_research_plus`;
- old imports are deprecated for removal;
- physical split repositories exist;
- plugins execute;
- live adapters are required.

## Release Notes

v1.0 docs should recommend new namespace facades for public API examples while
clearly saying `turing_research_plus` remains supported as the compatibility
namespace.
