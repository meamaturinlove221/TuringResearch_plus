I have been building **TuringResearch Plus**, a local-first Research OS for
evidence-led research workflows.

The problem I kept running into: research state gets scattered. Papers live in
one place, artifacts in another, route plans in notes, failures in chat logs,
advisor asks in separate documents, and claims start drifting away from the
evidence that should support them.

TuringResearch Plus turns that mess into a local, reviewable workflow:

- Evidence Ledger: separates planned work, fake/demo material, missing
  evidence, and observed evidence.
- Artifact Auditor: checks whether outputs are ready to support claims.
- Route DSL: records experiment routes, hard gates, blockers, and next actions.
- Paper Intelligence: supports digest, method cards, related work, deep review,
  and collision risk without fabricating final conclusions.
- Dashboard / Advisor Pack: packages project state into review surfaces.
- Plugin Safety: keeps plugin manifests, permissions, sandbox policy, and
  compatibility checks explicit.
- Privacy / Compliance Gates: keep public demos and case studies safe to
  review.

v1.0 is fake/demo-first by default. It has public demos, a VGGT public-safe
dogfooding case, campaign catalog, MCP fake/live config polish, modular
namespace facades, split-ready local bundles, and full regression checks.

What it does **not** do:

- it does not automate science;
- it does not write final papers;
- it does not replace advisors;
- it does not claim VGGT or SparseConv3D success;
- it does not run live adapters by default;
- it does not execute unknown plugins.

The project is partly a research workflow tool and partly an engineering
portfolio: contracts, tests, gates, local-first design, plugin safety, public
demo hygiene, and a clear refusal to overclaim.

If you care about research tooling, local-first systems, or honest AI-assisted
engineering workflows, this is the kind of project I want to keep developing.
