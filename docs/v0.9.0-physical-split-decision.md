# v0.9.0 Physical Split Decision

Status: planning decision.

Round: 167.

This document decides whether v0.9 should begin physical repository splitting.
It does not create repositories, move files, publish packages, or change
imports.

## Recommendation

Decision: `DEFER BROAD SPLIT / PREPARE CASE AND EXAMPLES ONLY`.

v0.9 should not split core runtime modules. The only split candidates worth
preparing for a possible physical split are:

1. `turingresearch-vggt-case`
2. `turingresearch-examples`

These candidates have the lowest risk of weakening the flagship repository.
They are presentation-heavy, public-demo-oriented, and can point users back to
the main repo without moving core implementation.

## Evaluation Matrix

| Dimension | Decision note |
| --- | --- |
| main repo star readiness | Keep the flagship repo as the primary star, install, docs, and demo entry point. |
| split candidate safety | Case/examples are safest after privacy, compliance, and claim-safety review. |
| docs completeness | Split design docs exist, but exact export manifests still need a final gate. |
| independent demo value | Case/examples have independent browsing value without owning core runtime. |
| maintenance cost | Case/examples add lower maintenance cost than runtime module splits. |
| interview value | Case/examples strengthen portfolio storytelling without making the main repo hollow. |
| community value | Examples help adoption; case study demonstrates dogfooding. |
| API stability | Core, paper, artifact, dashboard, and plugins are not stable enough for split. |

## Prioritize For Physical Split Planning

### 1. `turingresearch-vggt-case`

Reason:

- It demonstrates TuringResearch dogfooding.
- It can be public-safe if it contains only redacted case-study material.
- It does not need to own package APIs.
- It should point back to the main repo as the Research OS implementation.

Required before any actual split:

- exact export manifest;
- no private paths;
- no raw data;
- no SMPL-X model files;
- no unsupported VGGT or SparseConv3D success claim;
- privacy/compliance/claim-safety gate;
- maintainer approval.

### 2. `turingresearch-examples`

Reason:

- It can collect public demo projects and templates.
- It helps onboarding without extracting core code.
- It can be refreshed from the flagship with a sync policy.
- It can improve community value while preserving flagship star focus.

Required before any actual split:

- frozen public-demo export set;
- no real private logs;
- no API keys;
- no raw data or model payloads;
- exact test matrix;
- README links back to flagship.

## Defer

### `turingresearch-core`

Do not split core in v0.9. Core carries workspace, privacy, quality, template,
evidence, release, and compatibility semantics. Splitting it too early would
make the flagship repository feel hollow and harder to understand.

### `turingresearch-paper`

Defer until paper APIs are stable enough for users to install and test
independently. Paper writing and deep review still require explicit human
review and careful citation boundaries.

### `turingresearch-artifact`

Defer until artifact DTOs, remote artifact safety, handoff semantics, and
privacy/compliance gates are stable as a standalone package.

### `turingresearch-dashboard`

Defer until dashboard/export DTOs, optional PDF/PPTX backends, static assets,
and local-server roadmap are clearer.

### `turingresearch-plugins`

Defer until there is a real third-party ecosystem. The current plugin story is
manifest-first, disabled-by-default, and policy-gated.

## v0.9 Sprint Shape

Recommended v0.9 split work should be rehearsal-first:

1. freeze exact export manifests for case/examples;
2. add standalone README and safety gates;
3. run privacy, compliance, and claim-safety checks on exact export trees;
4. verify the main repo still has a complete demo;
5. decide whether to create real repos after maintainer review.

## Non-Claims

- This is not approval to split repositories.
- This is not approval to publish case-study claims.
- This is not approval to move core runtime code.
- This is not evidence that VGGT experiments succeeded.
- This is not a replacement for human release review.
