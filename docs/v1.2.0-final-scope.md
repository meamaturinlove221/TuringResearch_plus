# v1.2.0 Final Scope

Status: scope locked.

Round: 231.

v1.2 is locked as an original-reference parity release. The goal is to close
stable, practical gaps against the tracked reference repos before pursuing
newer ARIS-style research loops or speculative agent runtime work.

ARIS is valuable as a research direction, but it belongs in v1.3 study planning
or later. v1.2 should first make the existing TuringResearch line more complete,
testable, and reliable.

## Locked v1.2 Scope

1. Neocortica-Session parity.
2. Neocortica-Scholar parity.
3. Neocortica-Web parity.
4. yogsoth-ai campaign / vault / ontology parity.
5. MCP config parity.
6. Skill SOP parity.
7. Upstream strict diff.
8. Public demo refresh.
9. Full regression.
10. v1.3 ARIS study roadmap.

## What Parity Means

Parity does not mean copying upstream code. It means implementing or documenting
equivalent stable concepts in TuringResearch terms, with local-first defaults,
contracts, tests, and safety gates.

| Scope item | v1.2 interpretation |
| --- | --- |
| Neocortica-Session parity | Durable context handoff, pod lifecycle safety, return verification, shell/path/archive risk checks. |
| Neocortica-Scholar parity | Scholar pipeline docs/config polish, source fallback policy, citation status clarity. |
| Neocortica-Web parity | Web adapter docs/config parity, optional live mode clarity, Apify template planning without default live runs. |
| yogsoth campaign/vault/ontology parity | Campaign catalog, vault graph, ontology/SOP alignment, stress-test and experiment-planning continuity. |
| MCP config parity | `.mcp.example.json`, env block, fake/live defaults, no real keys. |
| Skill SOP parity | Skill entry/routing/SOP docs and tests aligned with stable public workflows. |
| Upstream strict diff | Reproducible local/manual snapshot policy with no invented live-diff claims. |
| Public demo refresh | Keep demos broad, fake/default, privacy-safe, and not VGGT-only. |
| Full regression | v1.2 release gate over v1.0/v1.1/v1.2 surfaces. |
| v1.3 ARIS roadmap | Study ARIS after parity, not inside the v1.2 implementation line. |

## Scope Relationship To Round 230

Round 230 identified useful v1.2 ecosystem candidates such as physical split,
docs deployment, dashboard polish, Pod Lifecycle Manager, and paper beta
refinement. Round 231 narrows the execution order: parity first, ecosystem
expansion second. The safest ecosystem items may remain planned, but the main
v1.2 implementation line should not chase new ARIS-style features before the
reference parity baseline is sound.

## Release Gates

Every v1.2 scope item must pass:

- name integrity;
- privacy/security gate;
- fake/live boundary check;
- no private data;
- no secrets;
- no raw data;
- no restricted model payload;
- no unsupported experiment success claim;
- no default networking;
- no unknown remote execution;
- no automatic child repo creation.
