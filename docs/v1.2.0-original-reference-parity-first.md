# v1.2.0 Original Reference Parity First

Status: policy.

Round: 231.

v1.2 prioritizes stable reference parity before ARIS-inspired expansion.

The tracked reference lines already point to practical, public-facing gaps:
context lifecycle safety, scholar/web MCP parity, campaign/vault/ontology
organization, skill SOPs, and strict upstream diff discipline. Those are more
important for v1.2 than adding cross-model review loops or meta-optimization.

## Why Parity First

- It reduces drift from the original references.
- It improves reproducibility and release discipline.
- It strengthens the local-first Research OS foundation.
- It gives v1.3 a cleaner base for ARIS study.
- It avoids mixing stable parity work with speculative agent runtime research.

## Reference Lines

| Reference area | v1.2 parity target |
| --- | --- |
| Neocortica-Session | context package lifecycle, transfer policy, return metadata verification, shell/path risk awareness |
| Neocortica-Scholar | MCP config style, scholar pipeline docs, source fallback policy, citation status clarity |
| Neocortica-Web | MCP web tool config style, optional live adapters, Apify template planning |
| yogsoth-ai | campaign routing, strategy catalog, vault/ontology/SOP patterns, stress-test and experiment-planning organization |

## ARIS Placement

ARIS is worth studying, but it should not enter v1.2 implementation scope.

Move to v1.3 study roadmap:

- ARIS cross-model review loop;
- ARIS meta-optimize;
- ARIS proof-checker;
- ARIS paper-writing automation;
- broader autonomous research orchestration.

## What Not To Do In v1.2

- Do not copy upstream code.
- Do not implement ARIS runtime loops.
- Do not turn paper beta into paper automation.
- Do not add unknown remote execution.
- Do not make live networking the default.
- Do not create child repositories automatically.
- Do not break the currently runnable mainline.

## Success Criteria

v1.2 succeeds if TuringResearch has:

- clearer reference parity reports;
- stricter upstream diff discipline;
- stronger MCP fake/live config parity;
- campaign/vault/ontology parity gates;
- session/pod lifecycle safety parity;
- refreshed public demos;
- full regression passing.
