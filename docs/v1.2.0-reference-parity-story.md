# v1.2.0 Reference Parity Story

Status: interview narrative.

Round: 253.

## Why Original Parity First

The original reference repositories already contained stable, practical ideas:
session context lifecycle, scholar/web configuration, skill-style SOPs,
campaign routing, vault and ontology review, stress testing, and experiment
runbook planning.

v1.2 prioritized those because they improve reliability immediately. They also
create a cleaner base for future research-agent ideas.

## What Was Aligned

| Reference area | TuringResearch v1.2 result |
| --- | --- |
| Session context | Context pack manifest, archive safety, structured return manifest, platform notes. |
| Scholar pipeline | Source priority, tool list export, MCP usage guide, fallback policy. |
| Web pipeline | Web fetching/content review surfaces, Apify optional-live guide, no-key skip. |
| MCP config | Clear env block, fake mode default, optional live flags, no real keys. |
| Skill SOP | Priority workflows now have when-to-use, inputs, outputs, safety, non-goals, handoff, tests, and docs. |
| Campaign catalog | Strategy book, preconditions, and review-only execution plans. |
| Vault and ontology | Wiki export, backlinks, edge audit, alias resolution, ontology gap detection. |
| Stress tests | Deterministic local review scenarios for common research workflow risks. |
| Experiment execution | Runbook and artifact requirements without automatic execution. |

## What Parity Does Not Mean

Parity does not mean copying upstream code. It means taking the stable concept,
adapting it to TuringResearch's local-first model, and putting it behind docs,
contracts, tests, and safety gates.

Parity also does not mean adopting heavy or risky parts:

- MinerU and heavy PDF fallback remain deferred.
- SSH, tmux, provision, and remote execution remain deferred.
- Unknown remote execution is rejected.
- Automatic experiment execution is rejected.

## Interview Framing

"I treated upstream references like product requirements, not code to copy. For
each feature I asked: what is the stable idea, what is the local-safe equivalent,
what should be tested, and what should remain deferred? That gives a much more
credible engineering story than simply adding flashy agent loops."
