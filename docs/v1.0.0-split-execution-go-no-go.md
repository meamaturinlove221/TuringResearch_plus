# v1.0.0 Split Execution Go/No-Go

Status: final v1.0 prelaunch split decision.

Round: 193.

This document decides whether TuringResearch Plus should physically create
split repositories before the v1.0 public launch. It does not create
repositories, push external remotes, move files, or change package install
paths.

## Decision

Decision: `NO-GO FOR PHYSICAL SPLIT BEFORE v1.0 PUBLIC LAUNCH`.

Decision: `GO FOR MAIN REPOSITORY PUBLIC LAUNCH PREPARATION`.

The main repository should launch first. Split-ready bundles may remain local
review artifacts until a maintainer explicitly approves real repository
creation.

## Priority After v1.0 Launch

1. `turingresearch-vggt-case`
   - First candidate after v1.0 launch.
   - Reason: strongest standalone story, public-safe dogfooding value, low risk
     to the flagship install path.
2. `turingresearch-examples`
   - Second candidate.
   - Reason: useful for onboarding and demos, but should follow the case repo
     so the flagship narrative remains concentrated.
3. `turingresearch-plugins`
   - Delay until real plugin contribution demand exists.
   - Reason: plugin policy is useful, but a standalone plugins repo without
     real third-party ecosystem may dilute attention and increase review cost.

## Not Split

The following are not physical split candidates for v1.0:

- core;
- paper;
- artifact;
- dashboard;
- plugin framework implementation.

They remain in the flagship because they define the product surface, public API,
install path, release gates, and safety boundaries.

## Rationale

- The main repository needs concentrated star and launch attention.
- Creating child repositories before launch adds README, license, issue,
  release, CI, and maintenance overhead.
- Split-ready bundles are enough to demonstrate future ecosystem shape.
- External links should not point to repositories that do not exist yet.
- Public feedback after v1.0 should decide whether the case/examples spokes
  are worth creating.
