# v1.0.0 Public API

Status: freeze draft.

Round: 176.

This document defines the v1.0 public API surface for TuringResearch Plus. It
does not move implementation code or remove the compatibility namespace.

## Compatibility Rule

`turing_research_plus` remains importable through v1.0.

The newer namespace packages are facade/re-export packages:

- `turing_research_core`
- `turing_research_paper`
- `turing_research_artifact`
- `turing_research_experiment`
- `turing_research_dashboard`
- `turing_research_plugins`
- `turing_research_cases`

Docs may recommend the new namespace facades, but v1.0 must not break existing
`turing_research_plus` imports.

## Stability Levels

| Level | Meaning |
| --- | --- |
| stable | Expected to remain compatible across v1.x except documented deprecations. |
| beta | Public enough to use, but may change with migration notes. |
| experimental | Useful and inspectable, but not a long-term integration promise. |
| internal | Not public API. May change without notice. |
| deprecated | Still available for compatibility, with a replacement path documented. |

## Module Public Surface

| Module | Namespace | v1.0 status | Public role |
| --- | --- | --- | --- |
| core | `turing_research_core` | beta with stable candidates | Workspace, privacy, quality, templates, evidence status semantics. |
| paper | `turing_research_paper` | experimental | Paper digest, method card, scaffold, related work, deep review. |
| artifact | `turing_research_artifact` | experimental | Artifact audit, handoff metadata, remote metadata, run ingest. |
| experiment | `turing_research_experiment` | beta with stable candidates | Route DSL, hard gates, failure reports, planned/executed boundary. |
| dashboard | `turing_research_dashboard` | experimental | Static dashboard, advisor export, export quality, vault/case views. |
| plugins | `turing_research_plugins` | experimental with stable manifest candidate | Plugin manifest, registry, MCP mapping, safety and capability reports. |
| cases | `turing_research_cases` | experimental | Public demos, case-study drafts, benchmark replay reports. |

## Stable Candidate Surfaces

These are candidates for v1.0 stability after final contract review:

- evidence status semantics;
- workspace/project summary DTOs;
- privacy report DTOs;
- quality/regression report DTOs;
- project template schema;
- route DSL summary schema;
- plugin manifest schema;
- benchmark/replay report schema.

## Public Import Examples

```python
from turing_research_core.public_api import Workspace, scan_privacy_paths
from turing_research_experiment.public_api import compile_experiment_route
from turing_research_plugins.public_api import PluginManifest, load_plugin_manifest
```

Compatibility imports still remain valid:

```python
import turing_research_plus
from turing_research_plus.workspace import Workspace
```

## Public API Rules

- Public API should expose DTOs, reports, manifests, and documented local tools.
- Public API should not expose internal helper modules.
- Experimental modules must not be described as stable.
- Deprecated APIs must list a replacement path.
- Imports must not require network access.
- Imports must not execute plugins.
- Optional PDF/PPTX/live backends must not be required by default.

## Boundaries

The v1.0 public API does not mean:

- automatic experiment execution;
- final paper writing;
- unknown plugin execution;
- OS-level plugin sandboxing;
- legal compliance approval;
- default live networking;
- physical split repositories.
